# 《天尊》资产与分镜字段向量化方案

本文档基于以下两份主表重新整理字段向量化方案：

- `《天尊》人设和场景表.xlsx`
- `《天尊》分镜1-19.xlsx`

目标不是把所有字段都塞进 embedding，而是区分：

- 哪些字段应该进入资产向量，用于“找角色、找场景、找道具、找造型”。
- 哪些字段应该进入分镜向量，用于“找某一镜、某句台词、某个动作、某个场景中的镜头”。
- 哪些图片应该先用视觉模型理解，生成文本描述后再向量化。
- 哪些字段只适合做过滤、回显、关联、审核或去重。

## 总体结论

建议拆成三条向量链路：

1. 资产向量：来自 `人设表`、`场景表`、`旧` 表，写入 `资产` / `资产变体`，用于资产检索。（也就是叶萧遥和叶萧遥升级版）
2. 分镜向量：来自 `分镜1-10`、`分镜11-19`，用于镜头检索和资产使用追溯。
3. 图片视觉语义向量：来自人设图、三视图、场景定稿图、分镜图、参考画面等图片，先用视觉模型生成文本描述、标签和质量信息，再向量化，用于“按画面找图”和“按图补全文本资产”。

不要把所有分镜文本直接拼进角色 `intro`。分镜是“使用数据”，角色和场景是“主数据”。二者应该关联，但不应该互相污染。

同理，不要直接把图片 URL 当成语义。图片需要经过视觉模型理解，得到可检索文本后再进入向量库。

## 人设表字段组织

`人设表` 的核心表头：

```text
姓名 / 简介 / 形象参考 / 外部参考图 / 人设图（定稿） / 三视图（按需） / 是否通过 / 首次出现集数 / 其他服饰
```

### 应进入资产向量

| Excel 字段 | 目标字段 | 向量化建议 |
| --- | --- | --- |
| `姓名` | `name` / `display_name` | 必须进入向量。角色名、别名、变体名都是核心召回词。 |
| `简介` | `intro` | 必须进入向量。承载身份、性格、背景、经历、关系、动机、结局。 |
| `形象参考` | `appearance` / `age_value` / `gender` / `height_cm` / `hair_description` / `outfit_description` | 必须拆分后进入向量。年龄、性别、身高也保留结构化字段。 |
| `其他服饰` | `outfit_description` 或变体描述 | 如果有文本，建议进入向量；如果是图片列、需要进入`asset_media`。但其视觉模型的语义理解需要入库时提前理解，并生成描述随资产一起入向量库。图片url不入 |

### 图片资产向量

| Excel 字段 | 建议用途 |
| --- | --- |
| `外部参考图` | `asset_media`，用视觉模型生成图片描述，进入图片视觉语义向量。 |
| `人设图（定稿）` | `asset_media`，作为主图，并用视觉模型理解角色外观、服装、姿态、画风。 |
| `三视图（按需）` |  `asset_media`，用视觉模型理解正面、侧面、背面、多视角造型差异。 |

### 不入资产向量
| `是否通过` | 审核过滤，对应 `approved` / `status`。 |
| `首次出现集数` | 结构化字段或 metadata，用于过滤、排序、追溯；一般不作为语义向量主体。 |

人设表中已经存在变体命名：

- `天尊版叶逍遥`
- `心魔叶婷`

建议继续按当前逻辑识别为 `asset_variants`：

| 原始名称 | 父主体 | 变体名 | 变体类型 |
| --- | --- | --- | --- |
| `天尊版 叶逍遥` | `叶逍遥` | `天尊版` | `look` |
| `心魔 叶婷` | `叶婷` | `心魔版` | `look` |

## 建议新增分镜向量文本

建议每一行分镜生成一条向量文本，格式类似：

```text
项目：天尊
集数：第一集
镜号：TJ1-2
场景：山中道台
出场角色：叶逍遥、叶婷
画面描述：叶逍遥盘坐道台，心魔叶婷缠绕在其身，叶逍遥双目紧闭
镜头：缓慢环绕推进
台词：...
音效：雷电轰鸣声，龙穿梭的声音
各种图片描述：。。。。
```

其中 `出场角色` 建议通过以下方式得到：

1. 从 `画面描述`、`台词`、`备注` 中匹配 `人设表` / `旧` 表里的角色名。
2. 对 `心魔叶婷`、`天尊版叶逍遥` 这类名称识别为变体，并关联父主体。
3. 如果一行只写了代词或称呼，先不强行补，避免误关联。

## 图片视觉语义向量

图片也需要进入语义检索，但路径应该是：

```text
Excel 内嵌图片
-> 提取并上传到 TOS
-> 写入 asset_media
-> 调用视觉模型理解图片
-> 生成结构化视觉描述
-> 将视觉描述文本 embedding
-> 写入图片视觉语义向量
```

这里的关键点是：向量化的不是图片 URL，而是视觉模型看图后生成的文本描述和标签。后续如果使用真正的多模态 embedding，也仍然建议保留这份文本描述，方便调试、回显、审核和人工修正。

### 图片描述如何反哺资产文本

视觉模型生成的图片描述可以反哺资产，但要做“轻量合并”：

- 如果 `hair_description`、`outfit_description`、`appearance` 为空，可以用主图描述补齐。
- 如果文本字段已有内容，图片描述只放入 `metadata.visual_understanding` 和图片向量，不要直接覆盖人工字段。
- 可以生成一个短字段 `visual_summary`，例如“主图显示为蓝白古风长袍、高马尾、仙侠气质”，进入资产向量。
- 多张图片不要全部拼进 `intro`，否则资产向量会变长且噪声变多。

## 分镜表对资产库的反哺

分镜表里能反哺资产库的信息很多，但建议以“关联”和“汇总”方式处理。

### 可以反哺到资产 metadata

| 来源 | 资产侧建议字段 | 用途 |
| --- | --- | --- |
| `集数` | `first_episode` / `episode_refs` | 角色或场景出现集数。 |
| `镜号` | `shot_refs` | 出现过的镜头。 |
| `场景` | `scene_usage_count` / `scene_aliases` | 场景使用频次和别名。 |
| 角色名匹配 | `mentioned_in_shots` | 角色出现镜头索引。 |

### 不建议直接合入资产向量全文

| 分镜字段 | 原因 |
| --- | --- |
| 全量 `画面描述` | 会让角色资产向量被具体镜头污染，且文本过长。 |
| 全量 `台词` | 台词适合镜头检索，不适合角色基础画像。 |
| 全量 `音效` | 音效属于镜头后期需求，不是资产本体。 |
| 制作状态和修改意见 | 属于生产管理数据，不是资产语义。 |

可以做一个短汇总进入资产侧，例如：

```text
使用摘要：第1集在山中道台渡劫，第2集在病房重生，第11-19集多次出现在豪华包厢和楚家别墅。
```

但这个汇总应该是后处理生成的短文本，不是直接拼所有分镜原文。
