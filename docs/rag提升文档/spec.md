# 中文在线短剧 Agent RAG 检索提升 Spec

## 背景

当前短剧资产智能体已经具备 Excel / JSON 资产入库、资产结构化字段写入 PostgreSQL、资产文本 embedding 写入 Milvus、语义检索并回表补详情的基础能力。

当前检索链路简单清晰，但仍处于“单路向量召回”阶段。对于短剧资产库来说，真实查询会同时包含角色名、别名、场景名、服装风格、人物关系、画面描述、台词、分镜上下文等多类意图。单纯依赖一条 embedding 召回链路，容易在精确名称、短 query、字段过滤、同角色多变体、图片视觉语义和分镜追溯场景下出现结果不稳定。

## 目标

- G1：形成一份面向当前项目的 RAG 检索提升方案，而不是泛泛介绍 RAG。
- G2：明确从低成本优化到中长期架构升级的分阶段路线。
- G3：让每一阶段都有可观察的收益指标和验收方式。
- G4：避免资产主数据、图片语义、分镜使用数据互相污染。
- G5：为后续编码实现提供可直接拆任务的设计依据。

## 功能需求

- F1：方案必须描述当前 RAG 链路和主要短板。
- F2：方案必须覆盖检索评测体系，包括 query 集、标注方式和指标。
- F3：方案必须提供混合召回策略，至少包含向量召回、关键词/精确匹配召回、融合排序。
- F4：方案必须提供二阶段重排策略，说明规则重排和模型重排如何逐步接入。
- F5：方案必须结合短剧资产领域，覆盖角色、场景、变体、图片视觉语义、分镜语义。
- F6：方案必须说明如何利用 metadata 过滤项目、资产类型、审核状态和复用范围。
- F7：方案必须给出阶段化实施顺序，区分短期、中期、长期。
- F8：方案必须引用网上调研资料，并说明这些资料如何映射到本项目。

## 非功能需求

- N1：方案不修改业务代码，不改变当前 API 行为。
- N2：方案应优先选择能渐进接入的设计，避免一次性大重构。
- N3：方案应兼容当前 FastAPI + PostgreSQL + Milvus + Doubao embedding 技术栈。
- N4：方案应便于后续拆分为 issue、任务或开发迭代。
- N5：方案中涉及对外接口的建议必须考虑向后兼容。

## 不做的事

- 不直接实现代码。
- 不替换当前 embedding 模型。
- 不直接引入新的生产依赖。
- 不设计完整前端交互。
- 不把所有分镜文本直接合入资产主体向量。
- 不把图片 URL 当成图片语义向量化。

## 验收标准

- AC1：文档中能看出当前系统是单路向量召回，并指出具体风险。
- AC2：文档中有明确的 P0-P4 或类似阶段路线。
- AC3：文档中有混合召回、RRF/融合、二阶段 rerank、query 改写、评测指标等方案。
- AC4：文档中能落到短剧资产语义：角色、场景、变体、分镜、图片视觉理解。
- AC5：文档中有可执行任务拆解和验收 checklist。
- AC6：文档中包含调研来源链接。
