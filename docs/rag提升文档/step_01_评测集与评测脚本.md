# Step 01：建立 RAG 检索评测集

## 这一步要解决什么

在改 RAG 检索逻辑之前，先建立一套固定评测方法。否则后面做混合召回、重排、查询改写时，很难判断效果是真的提升，还是只是在某几个测试词上看起来更好。

这一步的主文件不是 JSON，而是人能直接看的评测表：

```text
docs/rag提升文档/rag检索评测集.md
```

## 评测表已经写了什么

`rag检索评测集.md` 按检索场景拆成了几组：

| 分组 | 用来测什么 |
|---|---|
| 角色精确检索 | 直接输入角色名时，目标角色是否排第 1 |
| 场景精确检索 | 直接输入场景名时，目标场景是否排第 1 |
| 角色语义检索 | 不说名字，只说身份、关系、性格、剧情经历时能否找回角色 |
| 场景语义检索 | 不说场景名，只描述画面、功能、剧情地点时能否找回场景 |
| 变体与视觉描述检索 | 搜服装、状态、特殊形态、视觉特征时能否找到正确主体或变体 |
| 负样本和边界样本 | 搜明显不存在或跨类型的 query 时，结果是否过度自信 |

每一行都包含这些列：

| 列 | 说明 |
|---|---|
| `ID` | 评测样例编号，方便定位问题 |
| `Query` | 真实用户可能输入的检索词 |
| `应命中资产` | 理论上最应该返回的角色、场景或变体 |
| `应命中 ID` | 数据库里的真实资产 ID，用来避免只靠名字判断 |
| `资产类型` | `character`、`scene`、`variant` 或负样本 |
| `期望排名` | 目标资产应该进入 top 几 |
| `实际命中` | 跑接口后，实际最相关的命中结果 |
| `实际排名` | 目标资产实际排第几；没有出现就写 `未命中` |
| `实际分数` | 接口返回的 score |
| `是否通过` | 达到期望排名就写 `通过`，否则写 `失败` |
| `问题判断` | 失败时写原因，例如未召回、排序太低、类型混淆、变体混淆 |

## 手工怎么跑一条

以服务器接口为例：

```powershell
Invoke-RestMethod `
  -Method Post `
  -Uri "http://124.174.8.150:8000/api/v1/asset-search/semantic" `
  -ContentType "application/json; charset=utf-8" `
  -Body '{"query":"叶逍遥","limit":5}'
```

看返回里的 `items`，然后回到 `rag检索评测集.md` 对应行填写：

```text
实际命中 = top 结果里的 name
实际排名 = 期望 ID 在 items 里排第几
实际分数 = 期望 ID 对应的 score
是否通过 = 实际排名 <= 期望排名
问题判断 = 失败原因
```

## 判定规则

| 类型 | 通过标准 |
|---|---|
| 精确名称检索 | 目标资产排第 1 |
| 别名、括号名、同名批次 | 目标资产进入前 3 |
| 语义描述检索 | 目标资产进入前 5 |
| 场景描述检索 | 目标场景进入前 5 |
| 变体检索 | 优先命中变体；如果当前接口只返回主体，主体进入前 5 暂时算通过 |
| 负样本 | 不应该高置信返回无关短剧资产 |

## 为什么先写人工表

因为你现在最需要的是能看懂、能复盘、能和接口返回结果一一对上的验收表。自动化脚本可以后面再补，但第一版必须先让人看清楚：

```text
用户搜了什么
应该命中谁
应该排第几
实际命中谁
实际排第几
实际分数多少
为什么失败
```

有了这张表，再去做 topN 扩召回、关键词召回、RRF 融合、二阶段重排，才知道每一步到底有没有提升。
