# 当前检索方法与分数权重说明

更新时间：2026-07-09

本文档总结当前 `AssetVectorSearchService` 使用的检索流程、召回方式、融合排序方法和分数权重。对应代码主要在：

- `app/services/vector/asset_vector_search_service.py`
- `app/services/search/query_understanding_service.py`

## 1. 整体流程

当前检索不是单纯向量检索，而是：

```text
Query Understanding
-> 原始 query 向量召回
-> rewrite query 向量召回
-> 结构化 SQL 召回
-> 多路结果合并去重
-> RRF 融合排序
-> 同主体轻微降权
-> 0-1 展示分数校准
```

执行顺序：

1. 用户输入自然语言 query。
2. 调用 `QueryUnderstandingService`，让豆包 LLM 理解 query。
3. LLM 输出结构化字段和最多 2 条检索改写 query。
4. 使用原始 query 做一次向量召回。
5. 使用改写 query 分别做向量召回。
6. 使用 LLM 解析出的结构化字段做 SQL 召回。
7. 合并所有召回结果，并按 `source_table + source_id` 去重。
8. 使用 RRF 计算融合排序分。
9. 对同一主体重复出现的结果做轻微降权。
10. 将内部融合分转换成 0-1 的展示分数。
11. 返回最终 TopK。

## 2. Query Understanding

当前会让 LLM 把用户问题解析成数据库字段相关的 JSON。

主要字段：

| 字段 | 作用 |
|---|---|

| `asset_kind_hint` | 资产类型提示 |
| `name_hint` | 资产名称提示 |
| `display_name_hint` | 展示名称提示 |
| `intro_terms` | 简介关键词 |
| `appearance_terms` | 外观关键词 |
| `age_value_hint` | 年龄提示 |
| `gender_hint` | 性别提示 |
| `height_cm_hint` | 身高提示 |
| `hair_description_terms` | 发型/头发关键词 |
| `outfit_description_terms` | 服装关键词 |
| `category_terms` | 分类关键词 |
| `style_tags` | 风格标签 |
| `approved_hint` | 审核状态提示，当前解析但检索服务里暂未使用 |
| `reuse_scope_hint` | 复用范围提示，当前解析但检索服务里暂未使用 |
| `status_hint` | 状态提示，当前解析但检索服务里暂未使用 |
| `negative_terms` | 负向词，当前解析但检索服务里暂未使用 |

资产类型 `asset_kind_hint` 当前允许值：

```text
人像、妆造、服装、场景概念、场景四视图、站位图、道具、参考图、
视频片段、成片视频、音色、音乐、选景/堪景图、未知
```

注意：当前 `asset_kind_hint` 只参与 query 理解结果保存，现版本融合分里没有给资产类型匹配加分。

## 3. 向量召回

当前向量召回会使用这些 query：

1. 原始用户 query。
2. LLM 生成的第 1 条 rewrite。
3. LLM 生成的第 2 条 rewrite。

代码里会去重，避免相同 query 重复调用 embedding：

```python
search_queries = list(dict.fromkeys(item for item in search_queries if item))
```

每条 query 的内部召回数量：

```python
internal_limit = min(max(limit * 5, 20), 50)
```

含义：

- 如果用户要 `limit=10`，内部召回 `50` 条。
- 最少召回 `20` 条。
- 最多召回 `50` 条。

为什么不只召回用户要的 TopK：

- 后面还要合并多路结果。
- 后面还要重排。
- 如果第一阶段只召回 5 条，很容易漏掉正确答案。

## 4. 结构化 SQL 召回

当前 SQL 召回不是 BM25。

它是基于 LLM 解析出的字段做 PostgreSQL 条件召回，主要使用：

- `ILIKE '%关键词%'` 模糊匹配。
- `==` 精确匹配。
- `between` 范围匹配。
- 多个条件之间使用 `OR`。

当前 SQL 召回字段：

| LLM 字段 | SQL 匹配字段 | 匹配方式 |
|---|---|---|
| `name_hint` | `name`、`display_name` | `ILIKE` |
| `display_name_hint` | `display_name` | `ILIKE` |
| `source_project_name_hint` | `source_project_name` | `ILIKE` |
| `gender_hint` | `gender` | `==` |
| `age_value_hint` | `age_value` | 上下 3 岁 |
| `height_cm_hint` | `height_cm` | 上下 5cm |
| `intro_terms` | `intro` | `ILIKE` |
| `appearance_terms` | `appearance` | `ILIKE` |
| `hair_description_terms` | `hair_description` | `ILIKE` |
| `outfit_description_terms` | `outfit_description` | `ILIKE` |
| `category_terms` | `category` | `ILIKE` |
| `style_tags` | `style_tags` 转字符串 | `ILIKE` |

SQL 召回内部会先取：

```python
sql_fetch_limit = max(limit * 2, 40)
```

当前调用 SQL 召回时传入 `limit=20`，所以实际最多先从数据库取 `40` 条，再做结构化分数过滤和排序，最后返回最多 `20` 条。

SQL 召回保留门槛：

```python
MIN_STRUCTURED_SQL_SCORE = 0.12
```

也就是说，结构化字段匹配分低于 `0.12` 的候选会被丢弃，避免太宽泛的 SQL 条件污染最终排序。

## 5. SQL 结构化字段分

SQL 召回后，会用 `_score_structured_match` 重新计算一个结构化匹配分。

这个分数不是向量相似度，只表示“数据库字段和 query 结构化意图匹配得多不多”。

| 命中字段 | 加分 |
|---|---:|
| `name` 包含 `name_hint` | `+0.15` |
| `display_name` 包含 `display_name_hint` | `+0.10` |
| `source_project_name` 包含 `source_project_name_hint` | `+0.08` |
| `gender` 等于 `gender_hint` | `+0.06` |
| `age_value` 距离目标年龄不超过 3 岁 | 最高 `+0.08`，每差 1 岁减 `0.02` |
| `height_cm` 距离目标身高不超过 5cm | 最高 `+0.05`，每差 1cm 减 `0.005` |
| `intro` 命中一个 `intro_terms` | 每个 `+0.04` |
| `appearance` 命中一个 `appearance_terms` | 每个 `+0.05` |
| `hair_description` 命中一个 `hair_description_terms` | 每个 `+0.06` |
| `outfit_description` 命中一个 `outfit_description_terms` | 每个 `+0.06` |
| `category` 命中一个 `category_terms` | 每个 `+0.05` |
| `style_tags` 命中一个 `style_tags` | 每个 `+0.05` |

SQL 结构化分最高限制：

```python
return min(score, 0.75), matched_fields
```

这样做是为了避免 SQL 字段分压过向量召回。SQL 只作为补充召回和辅助排序信号，不作为唯一主分。

## 6. 多路结果合并

当前合并去重 key：

```python
key = (item["source_table"], item["source_id"])
```

也就是：

- 同一张表里的同一个 ID，只保留一个结果。
- 如果同一资产被原始 query、rewrite、SQL 多路召回命中，会合并它们的 `recall_signals`。

合并时展示字段会保留原始向量分更高的那份，但排序信号会合并。

这样做的原因：

- 如果同一个资产被多路召回命中，说明它更稳定相关。
- RRF 需要知道它分别在哪些召回通道里排第几。

## 7. RRF 融合排序

当前核心排序使用 RRF，也就是 Reciprocal Rank Fusion。

RRF 不直接相信不同通道的原始分数，而是看“每个候选在每一路召回里排第几”。

当前参数：

```python
RRF_K = 60
RRF_SCALE = 100.0
RECALL_SOURCE_WEIGHTS = {
    "vector_original": 1.00,
    "vector_rewrite": 0.85,
    "structured_sql": 0.70,
}
```

单个召回信号的 RRF 分数公式：

```text
rrf_score = source_weight * RRF_SCALE / (RRF_K + rank)
```

解释：

- `rank` 越靠前，分数越高。
- `RRF_K=60` 会平滑排名差异，避免第 1 名和第 2 名差距过大。
- `RRF_SCALE=100.0` 只是把内部融合分放大，方便和后续 bonus 放在一个量级。
- 原始 query 权重最高，因为它最忠实于用户输入。
- rewrite 权重略低，因为它是模型改写，有可能偏离。
- SQL 权重更低，因为它是字段包含/范围匹配，容易比较宽。

按当前参数，单路 rank=1 的基础分大约是：

| 召回来源 | 权重 | rank=1 时 RRF 分 |
|---|---:|---:|
| 原始 query 向量召回 | `1.00` | `1.6393` |
| rewrite 向量召回 | `0.85` | `1.3934` |
| SQL 结构化召回 | `0.70` | `1.1475` |

## 8. 额外融合加分

RRF 基础分之外，当前还有三个辅助项。

### 8.1 多路命中奖励

```python
multi_hit_bonus = 0.15 * min(len(recall_signals) - 1, 3)
```

含义：

- 如果一个资产只被一路召回，没有奖励。
- 如果被 2 路召回，加 `0.15`。
- 如果被 3 路召回，加 `0.30`。
- 如果被 4 路或更多召回，最多加 `0.45`。

为什么要加：

- 多个 query 或多个召回方式都命中同一资产，说明它不是偶然命中。
- 这个奖励比 RRF 主分小，只做稳定性补充。

### 8.2 SQL 结构化分奖励

```python
structured_bonus = structured_score * 0.40
```

含义：

- SQL 自己算出的结构化字段分不会直接作为最终分。
- 它只按 `40%` 的比例加入融合分。

例如：

- `structured_score=0.30`，最终加 `0.12`。
- `structured_score=0.75`，最终加 `0.30`。

为什么要乘 `0.40`：

- SQL 是辅助信号，不应该压过向量语义召回。
- 但如果字段确实命中，也应该帮助候选往前排。

### 8.3 名称匹配奖励

当前名称匹配只看：

- `name`
- `display_name`
- `parent_entity_name`

使用的 hint：

- `name_hint`
- `display_name_hint`

权重如下：

| 情况 | 加分 |
|---|---:|
| `name_hint` 或 `display_name_hint` 与名称完全一致 | `+0.35` |
| `name_hint` 或 `display_name_hint` 被名称包含 | `+0.25` |
| 原始 query 规范化后与名称完全一致 | `+0.20` |

为什么名称加分比较克制：

- 名称命中很重要，但不能完全覆盖视觉语义。
- 如果用户 query 是一长句，不能用整句去做强名称匹配，所以优先使用 LLM 提取的 `name_hint`。

## 9. 当前没有使用的加分项

之前讨论过的两个加分项，当前代码里没有再加：

| 项 | 当前状态 |
|---|---|
| `asset_kind_hint` 类型匹配加分 | 未使用 |
| `source_file_url` 有图加分 | 未使用 |

也就是说，当前排序主要由：

1. 多路召回排名；
2. SQL 结构化字段分；
3. 名称匹配；
4. 同主体去重降权；

共同决定。

## 10. 同主体重复结果降权

当前会对同一主体重复出现的结果做轻微降权。

逻辑：

```python
parent_id = item.get("parent_entity_id") or item.get("source_id")
if parent_id in seen_parent_ids:
    final_score -= 0.20
```

含义：

- 如果前面已经出现过同一个主体，后面的同主体结果减 `0.20`。
- 主要用于减少同一个角色或同一个资产的多个 variant 连续刷屏。

注意：

- 这个降权发生在 RRF 融合排序之后。
- 降权后会再排序一次。

## 11. 0-1 展示分数校准

RRF 原始融合分可能大于 1，不适合直接展示给用户。

当前会把内部融合分转换成 0-1 之间的相关性分数：

```python
RELEVANCE_SCORE_TEMPERATURE = 2.75
relevance_score = 1 - exp(-raw_score / 2.75)
```

含义：

- `raw_score` 是内部融合分，只适合排序。
- `relevance_score` 是对外返回的 `score`，范围是 `0-1`。
- 分数会保留 4 位小数。

为什么不用 `raw_score / max_score`：

- 如果按当前结果最大分归一化，第一名永远是 `1.0`。
- 这样看不出第一名本身到底是不是足够可信。

当前转换方式的特点：

- 多路高排名命中的结果会接近 `0.8-0.9`。
- 弱信号结果会自然低一些。
- 不会因为第一名存在就强行给它 `1.0`。

最终返回字段：

```python
item["score"] = relevance_score
```

内部原始分保存在：

```python
metadata["search_debug"]["raw_fusion_score"]
```

## 12. Debug 字段

当前每个结果的 `metadata.search_debug` 会保留排序解释信息。

常用字段：

| 字段 | 含义 |
|---|---|
| `recall_query` | 当前召回使用的 query |
| `vector_rank` | 在向量召回里的排名 |
| `recall_source` | 召回来源 |
| `understanding` | LLM query understanding 结果 |
| `fusion_score_before_diversity` | 同主体降权前的融合分 |
| `final_score` | 同主体降权后的内部融合分 |
| `fusion_reasons` | 每个加分/降权原因 |

`fusion_reasons` 里可能看到：

## 14. 当前方法的优点

1. 比单纯向量检索更稳，因为原始 query、rewrite、SQL 字段召回可以互补。
2. RRF 用排名融合，不强行相加不同通道的原始分，适合多路召回。
3. SQL 结构化分有上限和门槛，不会无限放大字段匹配。
4. 最终展示分是 0-1，更符合接口调用方理解。
5. Debug 信息比较完整，方便定位某个结果为什么排上来。

## 15. 当前方法的限制

1. SQL 召回不是 BM25，只是字段模糊匹配和范围匹配。
2. SQL 条件使用 `OR`，召回会偏宽，需要靠 `MIN_STRUCTURED_SQL_SCORE` 和后续重排过滤。
3. 中文同义词、隐喻、视觉语义主要依赖向量召回和 LLM rewrite。
4. 如果图片语义描述不准确，向量召回会受影响。
5. 如果 LLM query understanding 抽字段错误，SQL 召回会引入噪声。
6. 当前 `negative_terms` 已解析但还没有进入过滤和降权逻辑。
7. 当前没有真正的 cross-encoder reranker，最终仍然是规则化融合排序。

## 16. 当前最终排序公式汇总

可以把当前内部融合分理解成：

```text
raw_fusion_score =
    sum(source_weight * 100 / (60 + rank))
    + multi_recall_hit_bonus
    + structured_score * 0.40
    + name_match_bonus
    - same_parent_penalty
```

其中：

```text
multi_recall_hit_bonus = 0.15 * min(recall_signal_count - 1, 3)
same_parent_penalty = 0.20
```

最终对外展示分：

```text
score = 1 - exp(-raw_fusion_score / 2.75)
```

最终返回给接口调用方的 `score` 是这个 0-1 分数，不是 Milvus 原始向量分，也不是 SQL 结构化分。
